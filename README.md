## Baseline 
В качестве LLM выбрал deepseek-v4-flash. 
- Модель без few-shot примеров в промте выдала accuracy: 0.41 на 100 примерах, 0.46333 на 300. 
- Модель с few-shot  примерами в промте выдала accuracy: 0.39 на 100 примерах.
Я сделал вывод, что few-shot в данном случае только сбивает модель, ведь общего между разными запросами, даже с одинаковой меткой очень мало.<br><br>
Прочитал, что хорошо может повлиять динамический промт, few-shot для которого отбирается по косинусному сходству эмбедингов. Но реализовал я его уже у агента, этот подход выдал качество хуже, чем без few-shot и я решил дальше не продолжать эту идею (блоки кода я просто удалил). Я так понимаю - это из-за не очень большой выборки

## Agent
К LLM добивил инструмент поиска в интернете (duckduckgo)<br><br> 
Запрос агента в интернет строился без permalink, т.к. это ухудшит поисковые возможности. <br><br>
**Граф** агента получился такой:
1. LLM видит всю историю сообщений, решает, достаточно ли уже информации, или нужен поиск.
2. После этого результат передаётся в маршрутизатор - функцию should_continue, которая решает вызывать инструмент или переходить к finalize (просто набор условий).
3. Если необходим поиск - вызывается функци execute_tool, возвращающая результат поиска, затем обновляется состояние агента, переход к первому пункту.
4. Иначе - переход к функции finalize, в которой вызывается LLM с with_structured_output для получения итоговой оценки, формирования валидного ответа.<br><br>
